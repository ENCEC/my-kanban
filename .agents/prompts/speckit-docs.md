# Speckit 文档生成母提示词

你现在要为当前功能生成 speckit 文档。请严格遵守以下风格和规则。

## 总体目标

文档不是为了展示知识面，而是为了推动实现落地。请始终围绕“当前功能如何实现”来写，避免泛泛而谈、背景科普、教程化说明和空泛优缺点罗列。

## 语言规则

1. 全文默认使用中文。
2. 技术名词、框架名、库名、命令、路径、协议名、表名、字段名、接口路径、文件名保持原文。
3. 规范性词汇要自然中文化表达，但不要生硬翻译技术标识符。

## 整体写作风格

1. 风格偏工程决策纪要，而不是研究报告。
2. 先给结论，再给理由。
3. 少讲概念，多讲约束、取舍、影响和落地方式。
4. 只展开会影响架构、接口、数据模型、任务拆分和实现复杂度的内容。
5. 如果存在明显合理的默认方案，请直接给出推荐，不要平均展开所有候选项。
6. 避免大而全，优先简洁、清晰、可执行。
7. 不要使用空话，例如“便于未来扩展到企业级场景”，除非需求明确要求。

## 上下文使用规则

生成前请综合以下输入：

- 用户当前需求
- spec.md
- research.md
- plan.md
- data-model.md
- contracts/
- constitution.md

如果某些输入不存在，就基于已有信息做合理假设，并明确让输出内容与这些假设保持一致。

## 各文档阶段要求

### 如果目标是 `research.md`

- 将文档写成“Phase 0 Research / 技术决策记录”
- 不写泛泛综述
- 开头包含标题、Feature、Date、简短说明
- 主体按编号小节组织
- 每节只讨论一个关键决策
- 每节必须严格包含：
  - `Decision:`
  - `Rationale:`
  - `Alternatives considered:`
- `Rationale` 必须紧扣当前项目需求、约束、复杂度和开发成本
- `Alternatives considered` 不能省略，且要说明为什么不选
- 优先覆盖：
  - 前端框架与构建工具
  - UI / 拖拽 / 交互方案
  - 后端运行时与 API 组织
  - 数据存储方案
  - 必要时补充状态管理、测试策略、部署约束
- 不要使用大表格
- 篇幅以精炼为主

### 如果目标是 `plan.md`

- 将文档写成“实现方案规划”
- 不要复述 spec
- 重点写清：
  - 技术上下文
  - 架构边界
  - 模块拆分
  - 前后端协作方式
  - 数据流
  - 非目标
  - 风险点
- 明确哪些内容属于 frontend、backend、data、contracts、tests
- 重点说明“准备怎么实现”，而不是“需求是什么”

### 如果目标是 `data-model.md`

- 只描述当前功能需要的数据模型
- 不要过度设计
- 每个实体说明：
  - 目的
  - 核心字段
  - 字段约束
  - 与其他实体的关系
  - 状态流转（如适用）
- 若适合关系型数据库，要体现主键、外键、唯一约束、排序字段、时间戳、状态字段等
- 所有设计都要能映射回需求，不要写教材式定义

### 如果目标是 `contracts`

- 输出面向实现的接口契约
- 按资源或模块组织
- 每个接口至少说明：
  - `Purpose`
  - `Method + Path`
  - `Request params / body`
  - `Response body`
  - `Error cases`
- 如果存在拖拽重排、批量更新、部分更新、状态变更、成员分配等操作，要明确设计
- 优先保持接口简洁、一致、易实现
- 不要为了形式上的 REST 纯粹性增加不必要复杂度

### 如果目标是 `tasks.md`

- 输出可执行、可验证、具备依赖顺序的任务列表
- 按实现顺序组织，而不是按文档顺序组织
- 每条任务使用“动作 + 对象 + 结果”的表达方式
- 任务粒度要足够支撑直接开工
- 区分 frontend、backend、database、contracts、tests、integration
- 避免模糊任务，如“完善功能”“补充细节”
- 对复杂交互要拆细，例如拖拽排序持久化、状态变更、成员分配、乐观更新、错误回滚等

## 额外风格校准

请尽量贴近以下特征：

- 看起来像真实项目里的实现前文档，而不是 AI 综述
- 明确写“选什么”，也明确写“为什么不选别的”
- 能直接支撑后续编码，而不是停留在抽象层
- 语气克制、务实、工程化

## 执行要求

请直接输出目标文档内容，不要解释你将如何写，不要附加写作说明，不要输出提示词分析。
