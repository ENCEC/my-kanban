# 功能规格说明：团队 Kanban MVP

**Feature Branch**: `[001-team-kanban-mvp]`

**创建日期**: 2026-07-07

**状态**: Draft

**输入**: 用户描述："构建一个简单的团队 Kanban 工具作为 v1，用户可以创建项目、添加任务、给任务分配预置负责人，并在不同状态列之间拖动任务卡片；不做登录，用于验证核心流程是否跑通。"

## 用户场景与测试 *(mandatory)*

### 用户故事 1 - 创建项目并录入初始任务 (优先级: P1)

作为验证流程的团队成员，我可以创建一个项目并向其中添加任务，从而让团队拥有一个可实际操作的看板。

**优先级原因**: 没有项目和任务，就没有可测试的看板，也无法验证核心流程。

**独立测试方式**: 通过创建新项目、添加多个任务，并确认这些任务以初始状态出现在该项目看板中，即可完整测试该故事。

**验收场景**:

1. **给定** 当前不存在同名项目，**当** 用户输入项目名称并创建项目，**则** 该项目被保存，并出现在项目列表中。
2. **给定** 某项目已存在，**当** 用户向该项目添加一个带标题的任务，**则** 该任务出现在看板的默认起始状态列中。
3. **给定** 某项目下已有任务，**当** 用户打开该项目，**则** 看板只显示属于该项目的任务。

---

### 用户故事 2 - 分配任务负责人 (优先级: P2)

作为团队成员，我可以将每个任务分配给预先定义的某个人，从而在验证过程中清楚看到任务归属。

**优先级原因**: 负责人分配是需求中的核心流程之一，但前提是任务已经存在。

**独立测试方式**: 通过为任务选择一个预置成员，并验证任务卡片与任务详情中都一致显示该负责人，即可完整测试该故事。

**验收场景**:

1. **给定** 系统中已有预置团队成员，**当** 用户将其中一人分配给某个任务，**则** 该任务保存并显示该负责人。
2. **给定** 某任务已有负责人，**当** 用户更改负责人，**则** 原负责人被新负责人替换。

---

### 用户故事 3 - 在状态列之间移动任务 (优先级: P3)

作为团队成员，我可以在不同状态列之间拖动任务卡片，从而让看板以可视化方式反映任务进展。

**优先级原因**: 跨列拖动是证明 Kanban 交互可用的关键能力，但它依赖项目和任务已经存在。

**独立测试方式**: 通过将任务从一个状态列拖动到另一个状态列，并验证刷新或重新打开看板后其新状态仍然保留，即可完整测试该故事。

**验收场景**:

1. **给定** 某任务当前显示在看板上，**当** 用户将其拖动到另一个状态列，**则** 该任务出现在目标列，并从原列中移除。
2. **给定** 某任务已被移动到新的状态，**当** 重新加载该项目看板，**则** 该任务仍保留在最近一次选定的状态列中。

---

### 边界情况

- 当用户尝试使用已存在的名称创建项目时会怎样？系统应阻止重复项目名，并提示用户选择其他名称。
- 当用户创建任务时未分配负责人会怎样？任务应保持未分配状态，并继续显示在看板上，直到有人为其分配负责人。
- 当用户把任务拖回更早的状态时会怎样？系统应允许该操作，并正常更新看板。
- 当某个项目暂时没有任务时会怎样？看板应显示空的状态列，并提供清晰的入口来添加第一个任务。

## 需求 *(mandatory)*

### 功能需求

- **FR-001**: 系统必须允许用户通过提供唯一的项目名称来创建项目。
- **FR-002**: 系统必须显示可用项目列表，以便用户打开已有项目看板。
- **FR-003**: 系统必须允许用户向选定项目添加任务，且任务至少包含标题。
- **FR-004**: 系统必须在新任务首次添加时，将其放入默认起始状态。
- **FR-005**: 系统必须在第一版中支持一组固定的、预定义的任务状态列。
- **FR-006**: 系统必须仅显示属于当前所选项目的任务。
- **FR-007**: 系统必须提供一个预定义的团队成员列表，供用户选择为任务负责人。
- **FR-008**: 用户必须能够为任务分配一个预定义团队成员，并可在之后更新该分配结果。
- **FR-009**: 系统必须清晰显示每个任务当前的负责人，或明确标识该任务未分配负责人。
- **FR-010**: 用户必须能够直接在看板上将任务卡片从一个状态列移动到另一个状态列。
- **FR-011**: 系统必须持久保存项目、任务、负责人分配和状态变更，以便用户再次进入时看到最新看板状态。
- **FR-012**: 系统必须防止在同一工作区内创建重复的项目名称。
- **FR-013**: 系统必须允许任务在没有负责人的情况下存在。
- **FR-014**: 系统必须允许用户在任意可用状态列之间前后移动任务。
- **FR-015**: 系统必须在创建、分配或移动操作成功或无法完成时向用户提供反馈。

### 关键实体 *(include if feature involves data)*

- **Project**: 表示一个团队协作事项的看板容器，具有唯一名称，并包含一组任务。
- **Task**: 项目中的一个工作项，包含标题、当前状态、可选负责人，以及其所属项目关系。
- **Team Member**: 预先定义的人员，可被分配为某个任务的负责人。
- **Status Column**: 预定义的工作流阶段，用于按当前进展状态对任务进行分组。

## 成功标准 *(mandatory)*

### 可衡量结果

- **SC-001**: 用户能够在2分钟内完成一次完整核心流程：新建项目→添加至少一个任务→为任务分配负责人→将任务卡片拖动到另一状态列。
- **SC-002**: 100% 的已保存任务在创建后立即出现在正确的项目看板和正确的起始状态中。
- **SC-003**: 在验证测试中，95% 的任务负责人变更操作能在用户执行后 3 秒内反映到看板上。
- **SC-004**: 在验证测试中，95% 的任务状态移动操作能在首次尝试时使任务出现在目标列中。
- **SC-005**: 至少 90% 的端到端验证流程能够在无阻塞性错误的情况下完成“创建项目、添加任务、分配负责人、跨状态移动任务”这一完整流程。

## 假设

- 第一版服务于一个使用共享工作区的单一团队，而不是多个组织或租户。
- 用户认证、权限控制、邀请机制和审计历史不属于第一版范围。
- 团队成员列表通过配置或 seed data 预先定义，并可在 v1 的所有项目中复用。
- 状态列集合在 v1 中为预定义内容，不需要支持用户自定义。
- 本版本只需要验证核心 Kanban 流程；截止日期、评论、标签、文件附件、筛选等高级功能不在范围内。
